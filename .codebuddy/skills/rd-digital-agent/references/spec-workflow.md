# Spec Workflow（方案文档规范）

## 何时产出 spec 文档

以下情况建议在 `specs/<feature_name>/` 下生成正式文档：
- 新建一个后端服务或微服务模块
- 新增一个前端页面 + 对应后端接口
- 数据库表结构变更
- 跨 2 个以上子项目的改动
- 用户明确说"写文档"或"记录方案"

## 文档模板

### 1. requirements.md

```markdown
# <功能名> · 需求文档

## 背景与问题
<一句话描述为什么要做>

## 用户故事
- 作为 <角色>，我希望能 <功能>，以便 <价值>

## 验收标准（EARS）
- When <触发条件>，系统应 <行为>
- While <前提>，when <触发>，系统应 <行为>

## 约束
- 性能：<如 200ms 内响应>
- 安全：<如 需管理员权限>
- 兼容：<如 不影响现有 API>
```

### 2. design.md

```markdown
# <功能名> · 技术方案

## 架构概述
<用文字或 mermaid 描述模块关系>

## 数据库变更
<新表 / 新增字段 / 迁移计划>

## API 设计
<接口路径、请求/响应格式>

## 前端组件树
<页面 → 组件层级>

## 风险与权衡
- 风险：<描述> → 缓解：<措施>
```

### 3. tasks.md

```markdown
# <功能名> · 实施计划

- [ ] 1. <任务标题>
  - <具体步骤>
  - _验收：<EARS 标准>_

- [ ] 2. <任务标题>
  - ...
```

## 不写 spec 的情况

以下场景跳过 spec，直接 execute：
- 单文件 bug fix（无 API 变更）
- 文案/样式微调
- 已有清晰 task list 的简单 CRUD
- 用户说"不用写文档，直接改"

## Agent 行为约束

1. **每个阶段必须求确认**：brainstorm 方案 → 用户确认 → plan → 用户确认 → execute
2. **不猜测产品行为**：需求模糊时主动追问
3. **UI 变更走 ui-design 先行**：先出方案，再落地
4. **保持文档简短**：不追求完整性，追求可执行性
