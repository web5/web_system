# 架构原则（腾讯 + 字节实践对照）

## 微服务拆分原则

### 腾讯 TKE 模式
- 服务按业务域拆分（auth / user / content / payment）
- 每个服务有独立数据库
- 通过 API Gateway 统一入口
- 服务间用 gRPC/HTTP 通信
- 配置中心统一管理（七彩石）

### 字节模式
- 服务按功能模块拆分
- 强调服务无状态
- 使用 Protobuf 定义 API 契约
- 统一 RPC 框架
- CI/CD 自动化部署

### 本项目的判断标准

| 拆分条件 | 说明 |
|---------|------|
| 独立业务域 | 有独立的用户故事和业务价值 |
| 独立数据库 | 数据不与其他服务强耦合 |
| 独立部署 | 可单独发布、扩缩容 |
| 独立团队 | 可由独立团队维护 |

**反模式**：
- 按技术层拆分（如"前端服务"+"后端服务"）
- 微服务过小（一个功能一个服务）
- 共享数据库

---

## 数据库设计原则

1. **一服务一库**：每个微服务独享数据库
2. **3NF 为基础**：先规范化，再按查询模式优化
3. **索引先行**：设计表时同步设计索引
4. **避免外键**：用应用层保证数据一致性
5. **分表策略**：单表 >500w 行考虑分区/分表

---

## API 设计原则

1. **RESTful 优先**：资源命名用名词复数
2. **版本化**：`/api/v1/...` 或通过 Header
3. **统一错误格式**：`{ code, message, data }`
4. **分页规范**：`{ page, pageSize, total, items }`
5. **幂等性**：PUT/DELETE 保证幂等

---

## 前端架构原则

1. **组件驱动**：页面 = 组件组合
2. **状态上提**：共享状态放 Pinia store
3. **按需加载**：路由懒加载
4. **CSS 隔离**：scoped style + CSS 变量
